条款18：让接口容易被正确使用，不易被误用

Make interfaces easy to use correctly and hard to use incorrectly.

​ 客户使用你的接口但却没有获得它所预期的行为，那么这个代码就不该通过编译；如果代码通过了编译，它的作为就应该是客户所想要的。

​ 想开发一个“容易被正确使用，不易被误用”的接口，首先必须考虑到客户可能做出什么错误。

​ 看起来没问题，但是实际上客户使用时，很容易犯下两种错误：

  1. 以错误次序传递参数；
  2. 传递一个无效的月份与天数；

​ 许多客户端错误可以因为导入新类型而获得预防。在防范“不值得拥有的代码”上，类型系统很有用。因此让我们来导入wrapper类型来区别天数、月份和年份：

​ 预防客户错误的另一个办法是限制类型内什么事情可以做，什么事情不能做。常见的限制是加上const。另一个一般性准则“让类型容易被正确使用不容易被误用”的表现形式是：除非有好理由，否则应该尽量让你的类型的行为与内置类型一致。避免无端与内置类型不兼容，真正的理由是为了提供行为一致的接口。一致性非常关键。

  1. 好的接口很容易被正确使用，不容易被误用；
  2. “促进正确使用”的办法包括接口的一致性，以及与内置类型行为兼容；
  3. “阻止误用”的办法包括建立新类型、限制类型上的操作，束缚对象值以及消除客户的资源管理责任；

设计类犹如设计类型

​ 在C++中，你定义一个新类，也就定义了一个新类型。

​ 设计优秀类是一个艰巨的工作。几乎每一个类都要求你面对如下问题：

  1. 新类的对象应该如何被创建和销毁？
  2. 对象的初始化和对象的赋值有什么差别？
  3. 新类的对象如果被值传递意味着什么？
  4. 什么是新类的合法值？
  5. 你的新类需要配合某个继承图系吗？
  6. 你的新类需要什么转换？
  7. 什么样的操作符和函数对此类型而言是合理的？
  8. 什么样的标准函数应该驳回？
  9. 谁该取用新类的成员？
  10. 什么是新类型的未声明接口？
  11. 你的新类型有多么一般化？
  12. 你真的需要一个新类吗？

总结

  1. 类的设计就是类型的设计。

条款20：宁以pass-by-reference-to-const替换pass-by-value

Prefer pass-by-reference-to-const to pass-by-value.

​ 默认情况下，C++用值传递对象至函数。除非另外指定，否则函数参数都是以实际实参的副本为初值，而调用端所获得的也是一个函数返回值的拷贝。但也因此，有时候由于对象的拷贝构造，会使得值传递十分耗时。例如我们一个自定义类里定义了数十个字符串，那么我们如果只使用值传递，每次都需要对数十个字符串进行拷贝，这十分耗时。

​ 这种传递方式的效率高很多，它不需要使用构造函数与析构函数。同时，需要注意const是非常重要的。原先的函数是值传递，说明我们不希望修改原来的元素，const保护参数不被修改。

我不得不吐槽，真的这翻译就是依托，我越读越迷糊，不如自己理解。

​ 以引用方式传递还可以避免对象切割的问题。当一个派生类对象以值传递并被视为一个基类对象，基类的拷贝构造函数会被调用，使得其切割掉派生类特性，变成一个基类对象。解决该问题的方法就是使用引用传递。

​ 但有时候使用值传递也是可以的，但值传递的代价不昂贵的时候。一般而言，我们可以假定，使用值传递的唯一对象就是内置类型和STL迭代器和函数对象。至于其它的，尽量用常量引用代替值传递。

总结

  1. 尽量用常量引用传递代替值传递。可以提高效率并避免切割问题。
  2. 内置类型和STL迭代器和函数对象不必遵循上述规则。

条款21：必须返回对象时，不要返回引用

Don’t try to return a reference when you mush return an object.

​ 如果一个函数需要返回一个新对象，那么应按值返回对象，而不是返回引用。因为引用必须指向一个 已经存在并具有名称的对象。如果返回引用，引用的目标对象必须在函数调用前已经存在，否则就会产生悬空引用、未定义行为或逻辑错误。

总结

  1. 不要让函数返回指向局部对象、动态对象或静态对象的指针或引用，否则会导致对象失效或冲突。

条款22：将成员变量声明为private

Declare data memeber private.

​ 我们先来看看，为什么不采用public成员变量。

​ 语法一致性来看，如果成员变量不是public，那么客户访问对象的唯一方法就是通过成员函数。这样public每样东西都需要加上小括号进行调用，不容易出现问题。同时，使用函数可以让你对成员变量的处理有更精确的控制：我们可以通过函数来控制成员的读写权限。最后，通过函数访问可以更好实现封装：

​ 现在，我们考虑成员函数averageSoFar。实现它有两种方法：第一种，设计一个成员变量，函数只需返回即可；第二种，每次被调用时重新计算平均值。

​ 第一种做法，会使得类对象内存空间变大，但是因此函数调用非常高效；第二种做法与之相反。因此，这两种做法各有千秋，如果我们在不同机器上采用不同策略，我们只需修改averageSoFar成员函数的实现即可。将成员变量隐藏在函数接口的背后，可以为所有可能的实现提供弹性。封装十分重要！

​ 而protected成员变量论点也类似，当你使用protected，修改它那么使用它的所有派生类都将被破坏，这也是一个很大的修改开销。

总结

  1. 切记将成员变量声明为private。可以使得类具有弹性。
  2. protected并不比public更具封装性。

​ 哪种更优呢？面向对象法则要求数据应该尽可能被封装，然而与直观相反，成员变量比非成员变量函数封装性低。提供非成员函数可允许对类相关机能有更大的包裹弹性，导致较低的编译相依度，增加可延伸性。

​ 封装的核心在于尽量减少能看到和访问对象内部数据的代码数量：被封装得越多，能访问它的代码越少，对实现的修改就越安全、影响范围也越小；因此成员变量应设为 private，并在功能等价的情况下优先选择非成员非友元函数，以避免扩大对私有数据的访问面，从而提升类的可维护性与可演进性。

​ 但需要注意：第一，这个论述只适用于非成员非友元函数，友元函数对类成员访问权力和成员函数相同。第二，只因在意封装性而让函数成为类的非成员，并不意味它不可以是其它类的成员。

​ 不要把所有便利功能都做成类的成员函数，而应将不需要访问私有数据的功能设计为 non-member、non-friend 函数，并按功能拆分到不同头文件中；这样可以让核心类保持精简稳定，降低编译和依赖成本，提高封装性，同时使系统在未来更容易按需扩展，就像 C++ 标准库将核心类型与辅助功能分离那样。

总结

  1. 用非友元/非成员函数替换成员函数，可以增加封装性、包裹弹性和机能扩充性。

条款24：若所有参数都需要类型转换，则使用非成员函数

Declare non-member functions when type conversions should apply to all parameters.

​ 当一个函数的所有参数在调用时都可能需要进行类型转换时，应将该函数实现为非成员函数而不是成员函数。原因在于，成员函数只能对其隐含的 this 参数（即左操作数）要求已经是该类类型，无法对左操作数进行隐式类型转换；而非成员函数的所有参数地位对等，编译器可以对每个参数执行必要的类型转换，从而保证表达式在不同参数顺序下都能正常工作。因此，对于左右对称、且双方都可能参与类型转换的操作，使用非成员函数能提供更好的通用性和一致性。

总结

  1. 如果某个函数的所有参数都需要进行类型转换，那么将该函数声明为非成员函数。

​ 首先，如果swap缺省对类和模板类提供可接受的效率，那么不需要做任何事情。其次，如果缺省版本的swap效率不足，请尝试做如下事情：

  1. 提供一个public swap成员函数，让它高效地置换类型的两个对象值，这个函数绝不应该抛出异常；
  2. 在类或模板所在的命名空间提供一个非成员swap，并令它调用成员函数swap;
  3. 如果正编写一个类，而非模板类，为类特化std::swap，并令它调用swap成员函数；

​ 最后，如果你调用swap，请确定包含一个using声明式，以便让std::swap在函数内曝光。

​ 成员版swap绝不可抛出异常，那是因为swap的一个最好应用是帮助类提供强烈而异常安全性保障。

总结

  1. 当std::swap对你类型调用效率不高时，提供一个swap成员函数，并确定这个函数不抛出异常；
  2. 如果提供一个成员swap，也该特化一个非成员swap用来调用前者，对于非模板类，需要特化std::swap;
  3. 调用swap时应针对std::swap使用using声明式，然后调用那个swap并且不带任何命名空间资格修饰；
  4. 千万不要尝试在std内加入对某些std而言新的东西。
